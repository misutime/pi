## 4. SubAgent 相关的 TUI 改进：哪些是通用能力

我们的 subagent 实现了三层代码：核心层（#1）、扩展层（#2）、以及 TUI 集成。删除前两层后，TUI 层有三块代码需要分别评估。

### 4.1 启动信息中的 Tools 列表（#33）

```
[Tools]
  read, bash, edit, write, grep, find, ls, spawn_agent, websearch (inactive), webfetch (inactive)
  ── 展开后 ──
  read (builtin)
  bash (builtin)
  edit (builtin)
  spawn_agent (builtin:pix) (inactive)
  websearch (builtin:webtools) (inactive)
```

**这是通用改进**。它展示所有已注册工具的来源（builtin/user/project/path）和激活状态。
与 subagent 的唯一关联是 `spawn_agent` 会出现在列表中——删除 subagent 后它自然消失。

**结论：保留。** 不需要任何修改，删除 subagent 核心后 Tools 列表自动不再包含 `spawn_agent`。

### 4.2 启动信息中的 Agents 列表（#30）

```
[Agents]
  reviewer, planner, scout
  ── 展开后 ──
  reviewer  anthropic/claude-sonnet-4  [read, bash, grep]  代码审查  ok
  planner   openai/gpt-5-mini          [read, bash, find]   实现计划  [pending]
  scout     openai/gpt-5-mini          [read, grep, find]   代码侦察  error: missing ls
```

**这是 subagent 专属功能**。它展示 agent 配置、模型、工具 allowlist、以及校验状态（ok/error/pending）。依赖 `this.session.agents` / `agentPreflight` / `agentToolValidation`——这些 getter 在删除 subagent 核心后不再存在。

**但概念本身有价值**。pi-subagents 已有对等功能：

| 我们的实现 | pi-subagents 等价物 |
|-----------|-------------------|
| 启动时展示 agent 列表 + 状态 | `/subagents-doctor`（诊断）+ `/subagents-fleet`（运行时视图） |
| preflight 工具校验 | `/subagents-doctor` 含工具可用性检查 |
| agent 配置展示 | `subagent({ action: "list" })` |

pi-subagents 的诊断是**按需查看**（slash command），而我们的是**启动时主动展示**。如果希望保留"启动时即可见"的体验，后续集成 pi-subagents 后可以调用其内部 API 获取 agent 列表并在 `showLoadedResources()` 中展示。但这不是当前清理阶段要做的事。

**结论：删除。** 后续集成 pi-subagents 时可按需重新实现（调用 pi-subagents API 获取 agent configs，展示在启动信息中）。

### 4.3 `builtin` scope group（#32）

```
扩展分组：project → user → builtin → path
```

这是 `buildScopeGroups()` / `getScopeGroup()` 的扩展，新增 `"builtin"` 作为第四种 scope。被 Tools 列表、Extensions 列表、Skills 列表、Themes 列表共用。

**与 subagent 无关**。它是为内置扩展（webtools、github-tools）设计的分组展示。如果保留内置扩展加载机制（决策 E），这个分组就有意义。即使暂时没有内置扩展，保留分组逻辑也不产生副作用。

**结论：保留。** 与决策 E 联动。

### 4.4 preflight 调用 + 错误合并（#29 #31）

```typescript
// 启动时异步运行 preflight
void this.session.runAgentPreflight().then(...)

// 将 worker 扩展错误合并到诊断
const preflightErrors = this.session.agentPreflight?.extensionErrors ?? [];
```

**纯 subagent 逻辑**。`runAgentPreflight()` 启动 worker 进程检测工具可用性；错误合并将 worker 侧的扩展加载错误展示在 TUI 诊断中。

**结论：删除。**

| 改动 | 决策 | 说明 |

## 5. 需要你决策的项

### 决策 A：Web Tools 扩展（#3）

**当前状态**：`packages/extensions/webtools/` 提供 `websearch` + `webfetch`，支持 Firecrawl → Exa → Gemini → Jina → Readability 自动回退。

**选项**：

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| A1. 保留为内置扩展 | 继续作为 `packages/extensions/` 下的内置扩展 | 多 provider 回退是独特价值；代码质量好 | 需要维护；依赖 5 个外部 API |
| A2. 迁移为独立仓库 | 从核心仓库移出，作为独立可安装的 pi 扩展 | 减少核心仓库体积；独立发版 | 不再是"开箱即用" |
| A3. 删除，用 pi-web-access 替代 | pi-web-access 是 Nico Bailon 的 web 工具扩展，提供类似能力 | 跟随外部成熟工具 | pi-web-access 没有多 provider 回退；需要评估是否满足需求 |
| A4. 删除，用 MCP server 替代 | 通过 pi-mcp-adapter 连接 Jina/Firecrawl MCP server | MCP 生态 | 每次搜索需要两次 LLM 调用（先 search MCP tools，再调用）；无自动回退 |

**我的建议**：A1（保留）。多 provider 回退是这个扩展的核心价值，外部工具没有等价物。代码量小（~800 行），维护成本低。

### 决策 B：GitHub Tools 扩展（#4）

**当前状态**：`packages/extensions/github-tools/` 提供 5 个 GitHub 只读工具，gh CLI 优先 + REST API fallback，含 git remote 自动解析。

**选项**：

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| B1. 保留为内置扩展 | 继续维护 | git remote 自动解析是独特价值；gh CLI 比 MCP 更快更直接 | 代码量大（~1200 行）；GitHub MCP server 覆盖更全 |
| B2. 删除 API 层，保留 git remote 解析 + 作为 MCP server 的辅助 | 仅保留 `git-remote.ts`，其余用 GitHub MCP server | 减少维护；MCP server 由社区维护 | 增加 MCP server 依赖；需要两次调用（解析 remote → 调 MCP） |
| B3. 完全删除，用 GitHub MCP server 替代 | 通过 pi-mcp-adapter 连接 `@modelcontextprotocol/server-github` | 零维护；MCP server 覆盖完整 GitHub API | 失去 git remote 自动解析；MCP proxy 方式每次需搜索工具描述 |

**我的建议**：B1 或 B2。git remote 自动解析是真正实用的功能（用户只需说"查看当前仓库的 PR"），MCP server 无法做到。

### 决策 C：`_runSystemPromptPersisted` 快照机制（#18）

**当前状态**：在首次 LLM 调用前快照 system prompt + active tools + model info，首次有消息时持久化到 session JSONL（作为 `system_prompt`、`run_config`、`tools` 自定义条目）。

**功能**：让 session 文件记录"这次对话用的是什么 prompt、什么工具、什么模型"，方便事后审计和复现。

**选项**：

| 方案 | 说明 |
|------|------|
| C1. 保留 | 有用的审计能力 |
| C2. 删除 | 非必须功能，简化 diff，减少 agent-session.ts 复杂度 |

**我的建议**：C2（删除）。这个功能与 subagent 同时引入，目前没有其他调用方依赖它（只是写入 session 文件，没有读取方）。后续需要时可以再加。

### 决策 D：默认工具从 4 个改为 7 个（#19 #42）

**当前状态**：默认激活的工具从 `[read, bash, edit, write]` 变为全部 7 个内置工具（加了 `grep, find, ls`）。变更在 `agent-session.ts` 和 `sdk.ts` 两处。

**影响**：用户不指定 `--tools` 时，agent 可以自由使用 grep/find/ls 进行代码探索。

**选项**：

| 方案 | 说明 |
|------|------|
| D1. 保留 7 个 | 更实用的默认值，减少用户配置 |
| D2. 回退到 4 个 | 与上游一致，减少行为差异 |

**我的建议**：D1（保留）。给 agent 代码探索能力是合理的默认行为。如果不想默认全部，后续可以通过 settings 控制。

### 决策 E：内置扩展加载机制（#22 #27 #28）

**当前状态**：`loadBuiltinExtensions()` 扫描并加载 `packages/extensions/`（dev）或 `coding-agent/extensions/`（release）下的内置扩展。在 `resource-loader.ts` 中被调用。在 `interactive-mode.ts` 中有 `builtin` scope group 展示。

**如果删除 webtools 和 github-tools**：此机制没有内置扩展可加载，但代码框架保留。

**选项**：

| 方案 | 说明 |
|------|------|
| E1. 保留整套机制 | 空的加载管道，为后续重新添加内置扩展预留（如 pi-subagents、pi-mcp-adapter 作为内置扩展） |
| E2. 保留函数但清空体 | `loadBuiltinExtensions()` 返回 `{ extensions: [], errors: [] }` |
| E3. 完全删除 | 删除函数 + 调用点 + `builtin` scope group |

**我的建议**：E1 或 E2。如果下一步要将 pi-subagents 和 pi-mcp-adapter 作为内置扩展集成，这个加载机制正是需要的。建议 E2（保留空函数），后续集成时再填充。

### 决策 G：AI 包 session affinity 简化（#40 #45 #46 #47）

**当前状态**：上游用 `sessionAffinityFormat: "openai" | "openai-nosession" | "openrouter"` 三态模型。我们简化为：completions 统一 `sendSessionAffinityHeaders` 布尔（仅 OpenAI 格式），responses 统一 `sendSessionIdHeader` 布尔。

**影响范围**：`types.ts` 类型定义 + `openai-completions.ts` 实现 + `openai-responses.ts` 实现 + `model-registry.ts` schema + 4 个测试文件。**总计 ~10 个文件，+150/-300 行**。

**风险**：这是**对上游 API 的 breaking change**。如果未来合并上游新版本，session affinity 相关会有大量冲突。

**选项**：

| 方案 | 说明 |
|------|------|
| G1. 保留我们的简化版 | 代码更简洁，语义更清晰 |
| G2. 回退到上游的三态模型 | 减少与上游的差异，便于后续合并 |

**我的建议**：G1（保留）。上游的三态模型过度设计，我们的简化方向正确。合并冲突是可控的（冲突集中在 4 个文件）。

### 决策 H：write.ts 写入验证（#41）

**当前状态**：`write` 工具写入后立即 stat 验证文件落盘，如果不匹配则读取实际内容进行诊断。Windows 上含 VirtualStore 路径检测。

**功能**：防止静默写入失败（如 Windows UAC 重定向、云同步冲突）。

**选项**：

| 方案 | 说明 |
|------|------|
| H1. 保留 | 提高写入可靠性 |
| H2. 删除 | 简化 write 工具，减少复杂度 |

**我的建议**：H1（保留）。写入失败是最难排查的 bug 之一，验证机制代价很低（一次 stat 调用）。

### 决策 I：`shouldStopAfterTurn` hook（#42 中的 hook 部分 + #44）

**当前状态**：允许在 SDK 层设置精确的轮次预算控制。`agent.ts` 原生支持此 hook（上游已有 `ShouldStopAfterTurnContext` 类型），我们只是在 sdk.ts 中暴露了它。

**功能**：创建 AgentSession 时可以设置 `shouldStopAfterTurn` 回调，在每轮结束后判断是否停止。被 subagent executor 使用过，但能力本身是通用的。

**选项**：

| 方案 | 说明 |
|------|------|
| I1. 保留 | 通用的精确控制能力 |
| I2. 删除 | 没有当前使用者 |

**我的建议**：I1（保留）。这是 agent core 已经支持的接口，sdk 层暴露是合理的最小化改动（~5 行）。
