# Pi 开发与发布指南

## 项目结构

```
pi/
├── packages/
│   ├── ai/              @earendil-works/pi-ai          模型定义、provider 实现
│   ├── tui/             @earendil-works/pi-tui         终端 UI 组件
│   ├── agent/           @earendil-works/pi-agent-core   Agent 核心循环
│   ├── coding-agent/    @earendil-works/pi-coding-agent CLI 入口、工具、SDK
│   └── orchestrator/    @earendil-works/pi-orchestrator 多 Agent 编排（未发布到 npm）
├── scripts/             构建、发布、检查脚本
└── .github/workflows/   CI/CD
```

五个包**同版本号（lockstep）**，由 `scripts/sync-versions.js` 强制同步。

`coding-agent` 是 CLI 入口（`bin.pi = "dist/cli.js"`），依赖其余三个发布包。`orchestrator` 不在 npm 发布列表中。

## 构建

```bash
npm run build
```

实际执行：

```bash
cd packages/tui && npm run build        # tsgo -p tsconfig.build.json
cd packages/ai && npm run build          # generate-models + tsgo
cd packages/agent && npm run build       # tsgo
cd packages/coding-agent && npm run build # tsgo + chmod + copy-assets
cd packages/orchestrator && npm run build # tsgo + chmod
```

产物在各自 `dist/` 目录。`ai` 包额外生成 `models.generated.ts` 和 image models。

## 开发流程

```bash
# 1. 修改代码
# 2. 检查（lint + typecheck + shrinkwrap + 浏览器 smoke）
npm run check

# 3. 跑测试（coding-agent 包内）
cd packages/coding-agent && npx vitest run test/tools.test.ts
```

**提交前必须通过 `npm run check`。**

## 本地发布测试（无需发布到 npm）

```bash
npm run release:local -- --out /tmp/pi-local-release --force
```

脚本做的事：

1. `npm pack` 四个发布包 → tarball
2. 构建 Bun 二进制（当前平台）
3. 创建隔离目录，用 `file:` 协议安装 tarball
4. 生成 `pi` shim 入口

验证：

```bash
# Node 包
/tmp/pi-local-release/node/pi --help
/tmp/pi-local-release/node/pi -p "say ok"

# Bun 二进制
/tmp/pi-local-release/bun/pi --help
/tmp/pi-local-release/bun/pi -p "say ok"
```

**必须在仓库外测试**，否则会解析 workspace 依赖而非打包产物。

## Fork 适配清单

fork 到自己的 GitHub 仓库后，需修改以下内容才能独立构建和发布：

### 1. 修改 npm scope 和仓库地址

所有 `package.json` 中的 `@earendil-works/pi-*` → 你的 scope，例如 `@misutime/pi-*`。涉及文件：

```
packages/ai/package.json
packages/tui/package.json
packages/agent/package.json
packages/coding-agent/package.json
packages/orchestrator/package.json
```

同时修改各包的 `repository.url` 和内部依赖版本号。

`coding-agent` 的 `dependencies` 中也引用了 `@earendil-works/pi-*`，需要一并替换。

### 2. 修改源码中的 scope 引用

```bash
# 全局搜索替换
grep -r '@earendil-works/pi-' packages/ --include='*.ts' -l
```

主要是 import 语句和类型引用。

### 3. 修改发布脚本

`scripts/publish.mjs` 和 `scripts/local-release.mjs` 中有硬编码的 `packages` 数组和 `@earendil-works/pi-*` 名称，需要替换。

### 4. 配置 npm 发布

原项目用 **npm trusted publishing**（GitHub Actions OIDC），不需要 npm token。你需要：

- 在 npm 上创建你的 scope（如 `@misutime`）
- 在 npm 包设置中配置 trusted publishing，关联你的 GitHub 仓库
- 修改 `.github/workflows/build-binaries.yml` 中的 `environment: npm-publish` 对应的 GitHub Environment

或者不走 CI，手动发布：

```bash
# 构建
npm run build

# 逐包发布
cd packages/ai && npm publish --access public
cd packages/tui && npm publish --access public
cd packages/agent && npm publish --access public
cd packages/coding-agent && npm publish --access public
```

### 5. GitHub Actions

如果不需要 CI 发布，可以删除 `.github/workflows/` 中与发布相关的 workflow（`build-binaries.yml`、`ci.yml` 等），或修改其中的仓库引用和环境配置。

## 正式发布流程（原项目）

```bash
# 补丁版本（修复/新增）
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:patch

# 破坏性变更
PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:minor
```

脚本自动执行：

1. 检查工作区干净
2. 更新所有包的版本号 → `npm install --package-lock-only`
3. 将各包 `CHANGELOG.md` 的 `[Unreleased]` 替换为 `[x.y.z] - 日期`
4. 重新生成模型文件、shrinkwrap
5. 运行 `npm run check`
6. `git commit -m "Release vX.Y.Z"` + `git tag vX.Y.Z`
7. 添加新的 `[Unreleased]` 段 → 再次 commit
8. `git push origin main && git push origin vX.Y.Z`

**推送 tag 后 CI 自动触发 npm 发布**，无需本地 `npm publish`。

## 关键脚本速查

| 命令 | 用途 |
|------|------|
| `npm run build` | 构建所有包 |
| `npm run check` | lint + typecheck + 完整性检查 |
| `npm run release:local -- --out <dir> --force` | 本地发布测试 |
| `npm run release:patch` | 发布补丁版本 |
| `npm run release:minor` | 发布破坏性变更版本 |
| `node scripts/publish.mjs` | 手动 npm 发布（跳过已发布版本） |
| `node scripts/publish.mjs --dry-run` | 预览发布内容 |

## 注意事项

- `npm install` 必须带 `--ignore-scripts`，防止 lifecycle scripts 执行
- `packages/coding-agent/npm-shrinkwrap.json` 由 `generate-coding-agent-shrinkwrap.mjs` 管理，不要手动编辑
- `packages/ai/src/models.generated.ts` 由 `generate-models.ts` 生成，不要手动编辑
- `packages/coding-agent/install-lock` 由 `generate-coding-agent-install-lock.mjs` 管理
- pre-commit hook 会阻止 lockfile 变更提交，除非设置 `PI_ALLOW_LOCKFILE_CHANGE=1`
- Node.js >= 22.19.0
